# THE ROAD : 길 위의 질문
# 판매 가능 수준의 2D 방탈출형 인터랙티브 스토리 게임 제작 문서 v1

## 0. 문서 목적

이 문서는 현재 기획 중인 **2026 여름 수련회 방탈출 웹 프로그램**을 단순 프로토타입이 아니라, 외부 행사에 판매 가능한 수준의 **2D 방탈출형 인터랙티브 스토리 게임**으로 고도화하기 위해 필요한 요소와 작업 순서를 정리한 문서다.

현재 프로젝트의 주제는 다음과 같다.

```txt
Who do you say I am?
너는 나를 누구라 하느냐?
마태복음 16:15-16
```

현재 목표는 단순히 클릭으로 스토리를 보는 웹페이지가 아니다.

최종 목표는 다음에 가깝다.

```txt
현실 방탈출 퍼즐
+
웹 기반 2D 컷신 게임
+
캐릭터의 길 이동
+
숨겨진 십자가 동선
+
마지막 고백 엔딩
```

---

## 1. 현재 참고 이미지에서 가져와야 할 핵심

### 1-1. 스케치북 / 손그림 질감

참고 이미지의 핵심은 고해상도 실사풍 일러스트가 아니다.  
오히려 아래 감성이 중요하다.

- 손으로 그린 선
- 종이 질감
- 흑백 스케치
- 거친 명암
- 여백
- 작은 캐릭터
- 상징적 오브젝트
- 제한된 색 사용

따라서 현재 AI 생성 이미지처럼 컷마다 그림체가 달라지는 방식은 맞지 않는다.  
판매 가능한 수준으로 가려면 먼저 **아트 바이블**을 만들어야 한다.

### 1-2. 작은 캐릭터 중심의 공간감

참고 이미지에서 중요한 것은 캐릭터가 귀엽다는 것만이 아니라, 캐릭터가 **공간 안에 실제로 존재하는 느낌**이다.

필요한 요소:

- 캐릭터가 바닥 위에 서 있음
- 캐릭터가 오른쪽/왼쪽으로 이동함
- 문, 길, 불빛, 폭풍이 같은 공간에 있음
- 배경과 캐릭터의 시점이 맞음
- UI가 장면을 덮지 않음

현재 구현처럼 검은 화면 위에 UI 카드가 떠 있는 방식은 “게임 월드”가 아니라 “설명용 인터페이스”처럼 보인다.

### 1-3. 방향감각이 있는 길

가장 중요한 문제는 동선이다.

우리 기획에서는 캐릭터가 지나온 길이 마지막에 십자가로 드러나야 한다.  
그런데 진행 중에는 십자가 모양이 직접 보이면 안 된다.

따라서 필요한 것은 다음이다.

```txt
진행 중:
방향감각은 있음
하지만 전체 지도는 보이지 않음

최종 퍼즐:
지금까지의 방향 조각을 조합하면
십자가였음을 깨닫게 됨
```

즉, 숨겨야 할 것은 “전체 모양”이지, “방향 정보” 자체가 아니다.

현재 문제는 십자가가 숨겨진 것이 아니라, 동선 자체가 사라져서 마지막에 복원할 근거가 없다는 점이다.

## 2. 판매 가능한 수준의 기준

이 프로젝트가 내부 행사 1회용이 아니라 판매 가능한 수준이 되려면, 단순히 예쁜 화면만으로는 부족하다.

### 2-1. 게임성

필수 조건:

- 클릭만으로 진행되지 않아야 함
- 각 스테이지마다 현실 퍼즐 또는 웹 입력이 있어야 함
- 웹 화면은 현실 행동에 반응해야 함
- 힌트 시스템이 있어야 함
- 팀별 진행 상태가 저장되어야 함
- 운영자가 강제 진행/복구할 수 있어야 함
- 마지막 퍼즐이 전체 서사를 회수해야 함

현재 구조는 “스토리 뷰어”에 가깝다.  
판매 가능한 수준은 “상호작용형 방탈출 게임”이어야 한다.

### 2-2. 시각 완성도

필수 조건:

- 캐릭터 디자인 통일
- 배경 스타일 통일
- 문, 길, 불빛, 폭풍, 모닥불의 시각 언어 통일
- 자막 스타일 통일
- UI 패널 스타일 통일
- 스케치북/흑백/불빛 포인트 컬러 규칙 유지
- 컷마다 그림체가 바뀌지 않아야 함

권장 색상 규칙:

```txt
기본 세계:
흑백 / 회색 / 종이 질감

위험과 압박:
검은 선 / 거친 명암 / 차가운 청보라

부름과 임재:
따뜻한 노랑 / 주황 / 모닥불 색

금지:
초반부터 따뜻한 색을 여러 곳에 남발하는 것
```

### 2-3. 연출 완성도

필수 조건:

- 캐릭터가 실제로 걷는 느낌
- 문에 들어가고 다시 나오는 느낌
- 폭풍에 버티다 무너지는 느낌
- 무너진 뒤 빛을 다시 보는 감정 전환
- 빛을 향해 걷다가 뛰는 회복감
- 모닥불 앞에서 정적이 흐르는 느낌
- 최종 질문이 충분히 늦게 나오는 구성

현재처럼 텍스트가 장면을 설명하면 안 된다.  
장면 자체가 이야기를 말해야 한다.

### 2-4. 운영 안정성

판매 가능한 행사 프로그램은 현장에서 터지면 안 된다.

필수 조건:

- 새로고침해도 현재 진행 유지
- 팀별 진행 상태 저장
- 인터넷 불안정 시 로컬 진행 가능
- 운영자 강제 스킵 가능
- 정답 수동 통과 가능
- 힌트 사용 기록 가능
- 완료 시간 기록 가능
- 에셋 로딩 실패 시 fallback 화면 제공
- 사운드 재생 실패 시 무음으로 진행 가능

### 2-5. 저작권 / 라이선스

판매 가능한 수준이라면 BGM, 폰트, 이미지, 캐릭터 에셋에 대한 권리를 정리해야 한다.

특히 실제 찬양 음원을 사용하거나, 행사 녹화/온라인 공개/유료 패키지로 배포하려면 별도 권리 확인이 필요하다.

권장 방식:

```txt
1차 내부 행사:
교회 내 사용 범위 확인

판매용 패키지:
저작권 문제 없는 자체 BGM 제작
또는 사용 허락받은 음원 사용

폰트:
상업적 이용 가능한 무료/유료 폰트만 사용

이미지:
직접 제작, 외주 제작, 또는 상업적 이용 허용 AI/스톡만 사용
```

## 3. 우리가 원하는 최종 장르 정의

이 프로젝트는 일반 웹앱이 아니다.

정확한 장르는 다음이다.

```txt
스케치형 2D 방탈출 컷신 게임
```

또는

```txt
현실 연동형 2D 인터랙티브 스토리 퍼즐 게임
```

### 아닌 것

```txt
검은 화면 텍스트 어드벤처
클릭형 웹툰
단순 방탈출 정답 입력창
React UI 카드 게임
텍스트 로그 뷰어
미니 설교 페이지
```

### 맞는 것

```txt
작은 캐릭터가 길을 걷는 게임
현실 퍼즐을 풀어 문을 여는 게임
문 안에서 겪는 상황이 컷신으로 보이는 게임
진행 중 모은 조각으로 마지막 동선을 복원하는 게임
마지막에 그 길이 십자가였음을 깨닫는 게임
그 후 “너는 나를 누구라 하느냐?” 앞에 서는 게임
```

## 4. 핵심 게임 루프

판매 가능한 수준의 구조는 다음 루프를 가져야 한다.

```txt
웹 컷신
→ 현실 단서 안내
→ 현실에서 퍼즐 탐색
→ 웹에 정답 입력
→ 웹이 반응
→ 문/길/캐릭터 상태 변화
→ 경로 조각 획득
→ 다음 스테이지
```

각 스테이지는 이 루프를 가져야 한다.

### 예시

```txt
컷신:
캐릭터가 낡은 문 앞에 멈춘다.

웹 지시:
“문은 열리지 않았다.”

현실:
팀은 실제 봉투/상자/카드에서 단서를 찾는다.

퍼즐:
조각난 문장 복원

웹 입력:
확신

웹 반응:
문이 조용히 열린다.

컷신:
캐릭터가 문 안으로 들어간다.

보상:
fragment_01 획득
```

## 5. 전체 스테이지 구조

최종적으로는 5개 스테이지 + Final 구조를 추천한다.

# STAGE 0. 길 위의 시작

## 목적

세계관 진입.  
캐릭터와 멀리 있는 빛을 소개한다.

## 시각

- 흑백 길
- 아주 먼 작은 주황빛
- 작은 캐릭터
- 넓은 여백
- 아직 문 없음
- 아직 모닥불 없음
- 기다리는 인물 없음

## 상호작용

- 팀 코드 입력
- 시작 버튼
- 기본 조작 안내

## 핵심 금지

초반에 모닥불을 보여주면 안 된다.  
초반에는 “불빛”만 존재해야 한다.

---

# STAGE 1. 첫 번째 문 — 익숙하지만 멀어진 신앙

## 목적

캐릭터가 첫 번째로 길에서 벗어나는 구간.

## 시각

- 조용하고 낡은 문
- 회색, 오래됨, 차분함
- 화려하지 않음
- 문 안은 비어 있고 조용함
- 익숙한데 낯선 분위기

## 문 안 상황

이 문은 “교회를 안 믿는 상태”가 아니다.  
오히려 신앙이 너무 익숙해졌지만 마음은 멀어진 상태다.

표현:

```txt
예배는 있었지만 마음은 없음
기도는 있었지만 확신은 없음
말씀은 들었지만 방향은 없음
```

## 현실 퍼즐 후보

- 조각난 문장 복원
- 익숙한 신앙 단어 중 빠진 단어 찾기
- UV 라이트로 숨겨진 단어 확인
- 성경 구절 직접 암기가 아니라, 문맥을 복원하는 퍼즐

## 웹 입력 후보

```txt
확신
FAITH
돌아감
```

## 경로 보상

```txt
fragment_01
```

겉보기에는 단순한 긁힌 선 조각이다.  
아직 경로 조각이라는 것을 알면 안 된다.

---

# STAGE 2. 두 번째 문 — 증명의 방

## 목적

세상이 주는 이름과 증명을 따라가는 구간.

## 시각

- 차갑지만 매력적인 문
- 청보라 포인트
- 네온처럼 보이지만 따뜻하지 않음
- 판타지 게이트 금지
- 문 안은 압박감 있는 실내

## 문 안 상황

이 방은 “유흥”을 직접 보여주는 곳이 아니다.  
핵심은 **나를 증명하라고 요구하는 세계**다.

표현:

```txt
성과
스펙
연봉
비교
인정
실패
불안
외모
팔로워
자극
```

## 시각 장치

- 떠다니는 종이
- 타오르는 이름표
- 차가운 네온 선
- 고개 숙인 캐릭터
- 점점 작아지는 캐릭터
- 소음 같은 선

## 현실 퍼즐 후보

- 단어 카드 분류
- 진짜 이름이 아닌 것 제거
- 숫자/색깔 조합
- 비교/성과 단어 카드에서 숨겨진 코드 추출

## 웹 입력 후보

```txt
이름이 아니다
NOT NAME
NOT MY NAME
```

## 경로 보상

```txt
fragment_02
```

---

# STAGE 3. 폭풍 — 무너짐과 다시 들리는 음성

## 목적

캐릭터가 더 이상 스스로 걸을 수 없게 되는 구간.

## 시각

- 길 위에서 멈춤
- 억지로 걷기
- 추상적 폭풍
- 바람선, 압박선, 노이즈
- 실제 토네이도 금지
- 무너짐
- 멀리 아주 희미한 빛 재등장

## 문 안 상황과 차이

Stage 2가 “세상 속 압박”이라면, Stage 3은 그 결과로 생기는 **내면 붕괴**다.

## 현실 퍼즐 후보

- 노이즈 속 반복 문장 찾기
- 여러 문장 중 계속 반복되는 한 문장 찾기
- 메시지/기도문 카드 작성
- 특정 문장을 실제 상자에 넣고 웹에서 전송 버튼 누르기

## 웹 입력 후보

```txt
여기 있다
STILL HERE
돌아오라
```

## 컷신 반응

- 캐릭터가 고개를 든다
- 멀리 있던 작은 빛을 처음으로 인식한다
- 불빛은 조금 더 선명해진다
- 아직 모닥불은 보이면 안 된다

## 경로 보상

```txt
fragment_03
```

---

# STAGE 4. 돌아가는 길 — 경로 회수

## 목적

처음부터 있었던 빛을 향해 돌아가는 구간.  
그리고 마지막 동선 퍼즐을 푸는 구간.

## 시각

- 캐릭터가 천천히 일어남
- 빛을 향해 걷기 시작
- 점점 달림
- 빛이 커지지만 모닥불 형태는 아직 금지
- 전체 지도는 보여주지 않음

## 핵심 퍼즐

질문:

```txt
걸어온 길은 어떤 모양인가?
```

이 질문은 이 스테이지에서 처음 등장해야 한다.

## 퍼즐 방식

지금까지 얻은 fragment_01~03과 마지막 fragment_04를 조합한다.

진행 중에는 조각들이 단순 긁힌 선, 문양, 흔적처럼 보였다.  
하지만 여기서 조합하면 “경로 조각”이라는 사실이 드러난다.

## 정답

```txt
십자가
CROSS
THE CROSS
```

## 핵심 조건

이 퍼즐 전까지는 절대 십자가 모양을 보여주면 안 된다.  
단, 플레이어가 “아, 우리가 이런 방향으로 움직였구나”라고 복원할 근거는 있어야 한다.

---

# STAGE 5. 모닥불 — 기다림

## 목적

처음부터 기다리고 계셨던 존재와 만나는 구간.

## 시각

- 여기서 처음 모닥불이 명확히 등장
- 기다리는 인물이 이미 앉아 있음
- 얼굴 없음
- 뒷모습/실루엣
- 작은 캐릭터가 조심스럽게 멈춤
- 캐릭터가 옆에 앉음
- 그분이 담요를 건네거나 자리를 내어줌
- 정적

## BGM

BGM은 여기서 바로 크게 틀지 않는다.

추천 순서:

```txt
모닥불 도착
→ 기다리는 인물 확인
→ 캐릭터가 옆에 앉음
→ 자상한 행동
→ 2~3초 정적
→ BGM 아주 작게 시작
→ 최종 질문
```

## 6. 방향감각 설계

이 프로젝트에서 가장 어려운 부분은 방향감각이다.

진행 중에는 십자가를 숨겨야 하지만, 마지막에 복원 가능해야 한다.

### 잘못된 방식

```txt
방향도 숨김
동선도 숨김
전체 지도도 없음
마지막에 갑자기 “무슨 모양?” 질문
```

이러면 플레이어는 풀 수 없다.

### 올바른 방식

```txt
전체 지도는 숨김
하지만 방향 경험은 남김

문으로 빠졌다
돌아왔다
다른 문으로 갔다
다시 돌아왔다
빛을 향해 갔다

이 경험이 나중에 조각과 함께 복원됨
```

### 구현 방법

#### 내부 route trace

게임 내부에서는 실제 동선을 기록한다.

```ts
type RouteDirection =
  | "FORWARD"
  | "LEFT_BRANCH"
  | "RETURN_MAIN"
  | "RIGHT_BRANCH"
  | "RETURN_SECOND"
  | "TOWARD_LIGHT";
```

#### 플레이어에게 보이는 방식

플레이어에게는 방향을 직접 지도처럼 보여주지 않는다.

대신 다음으로 보여준다.

- 문에 들어갈 때 캐릭터가 화면의 옆 방향으로 이탈
- 문에서 나올 때 다시 길로 복귀
- 같은 먼 불빛이 계속 같은 방향에 희미하게 존재
- Stage 완료 시 이상한 선 조각 획득
- 마지막에 조각을 맞추면 동선이 복원됨

## 7. 자막 설계

현재 자막 문제는 “스토리 설명문”이 그대로 올라온다는 것이다.

자막은 설명이 아니라 감정과 단서여야 한다.

### 금지 자막

```txt
캐릭터는 문 앞에서 망설인다.
캐릭터는 폭풍 속에서 무너진다.
캐릭터는 다시 빛을 발견한다.
현실 단서를 찾아 정답을 입력하세요.
```

### 권장 자막

```txt
익숙한 문이었다.
그래서 더 낯설었다.

많은 소리가 있었다.
내 이름은 없었다.

더는 걸을 수 없었다.

그 빛이,
아직 있었다.

문은 열렸지만
마음은 닫혀 있었다.

돌아갈 곳이 아니라
기다리는 곳이었다.
```

### 자막 규칙

```txt
한 줄 18자 내외
최대 2줄
설명문 금지
감정/질문/단서 중심
UI 지시는 자막이 아니라 작은 prompt로 분리
```

## 9. 최종 Route Puzzle 설계

이 프로젝트의 가장 중요한 퍼즐이다.

### 목표

플레이어가 지금까지 얻은 조각을 맞추며 깨달아야 한다.

```txt
우리가 지나온 길이 십자가였구나.
```

### 진행 중에는 숨기는 것

- 전체 지도
- 십자가 형태
- 탑뷰
- 노골적인 수직/수평 선 조합
- “십자가”를 연상시키는 직접 힌트

### 진행 중에는 남기는 것

- 방향 경험
- 문으로 빠졌다 돌아온 느낌
- 반복되는 먼 불빛
- stage별 fragment
- 조각의 선 흔적
- 문 안에서 나온 후 길로 복귀하는 컷

### 최종 퍼즐 UI

Stage 4에서 처음 route puzzle이 열린다.

구성:

```txt
낡은 조각 4개
각 조각에는 희미한 선이 있음
조각을 클릭하면 회전
조각을 배치하면 하나의 길처럼 연결
정답 입력창
```

MVP에서는 drag & drop까지 필수는 아니다.  
하지만 조각을 맞추는 느낌은 있어야 한다.

허용 정답:

```txt
십자가
CROSS
THE CROSS
cross
```

## 10. 기술 구조

현재 기술 방향은 유지한다.

```txt
Frontend:
Phaser 3 + TypeScript + Vite

선택:
Rive

Backend:
FastAPI + SQLite

초기 저장:
localStorage
```

Phaser 3와 Vite + TypeScript 조합은 HTML5 2D 게임 개발과 빠른 개발 서버/빌드에 적합하다.  
Rive는 State Machine으로 인터랙티브 애니메이션 상태를 제어할 수 있으므로, 나중에 캐릭터 걷기/무너짐/일어남/앉기 같은 고급 애니메이션에 제한적으로 쓰면 된다.  
FastAPI는 게임 화면이 아니라 팀 상태 저장, 정답 검증, 힌트 기록, 관리자 기능에 사용한다.

## 16. Codex / 개발 AI 의뢰용 요약 프롬프트

```txt
너는 Phaser 3 + TypeScript 기반 2D 방탈출형 인터랙티브 스토리 게임을 개발하는 시니어 게임 개발자다.

현재 프로젝트는 retreat-road-game이다.
현재 구현은 검은 배경 + 텍스트 + 카드형 UI 중심이라 우리가 원하는 방향과 다르다.

우리가 원하는 것은 다음이다.

“작은 미니멀 캐릭터가 흑백 스케치형 세계 안에서 길을 걷고,
현실 퍼즐을 풀어 문을 열고,
문 안에서 신앙의 공허함과 세상의 증명 압박을 지나고,
폭풍 속에서 무너졌다가,
처음부터 있던 빛을 다시 보고,
지금까지 걸어온 길의 조각을 맞춰 십자가였음을 깨닫고,
마지막에 모닥불 앞에 기다리던 존재 곁에 앉은 뒤
‘WHO DO YOU SAY I AM?’ 질문 앞에 서는 게임.”

핵심 요구:
1. 텍스트 뷰어가 아니라 스케치형 2D 게임 월드로 만들어라.
2. UI 카드는 줄이고 장면 자체가 스토리를 말하게 하라.
3. StoryNode 기반 구조로 리팩터링하라.
4. 각 Stage마다 cutscene → real-world interaction gate → answer input → success cutscene 구조를 넣어라.
5. RouteTraceManager를 만들어 stage별 경로 조각을 저장하라.
6. 진행 중에는 십자가 형태를 절대 보여주지 말라.
7. Stage 4에서만 “걸어온 길은 어떤 모양인가?” route puzzle을 열어라.
8. 정답은 십자가 / CROSS / THE CROSS이다.
9. Stage 5 전까지 모닥불과 기다리는 인물을 보여주지 말라.
10. Final Question은 모닥불 정적 이후에만 등장시켜라.
11. 자막은 설명문이 아니라 짧은 감정/단서 문장으로 바꿔라.
12. fallback graphics도 단순 도형이 아니라 손그림 스케치 느낌으로 그려라.
13. Phaser + TypeScript + Vite 구조를 유지하라.
14. React는 사용하지 말라.
15. localStorage에 현재 node, 힌트 사용, route fragment를 저장하라.

필수 신규 모듈:

- 모든 진행이 클릭만으로 끝나면 실패다.

- 진행 중에는 십자가를 숨기되, 마지막에는 동선과 조각으로 복원 가능해야 한다.
```

## 17. 최종 판단

지금 프로젝트의 핵심 문제는 기술이 부족한 것이 아니다.

문제는 구현 목표가 너무 낮게 잡혀 있었다는 점이다.

현재 목표:

```txt
컷을 넘기는 웹 프로토타입
```

이어서는 판매 가능한 수준이 되기 어렵다.

앞으로 목표는 이렇게 바꿔야 한다.

```txt
스케치형 2D 방탈출 컷신 게임 엔진
```

가장 먼저 해야 할 일은 예쁜 에셋 추가가 아니라 다음이다.

```txt
1. StoryNode 구조로 리팩터링
2. 스테이지별 interaction gate 추가
3. 방향감각과 route fragment 시스템 추가
4. Stage 4 route puzzle 구현
5. 스케치형 월드 렌더러 구현
6. 자막 문체 전면 수정
7. 문 안 상황과 모닥불 엔딩 고도화
```

이 순서대로 가야 현재 40점 미만의 프로토타입을 판매 가능한 수준으로 끌어올릴 수 있다.

## 18. 참고한 기술 근거

- Phaser 3 + Vite + TypeScript는 HTML5 게임 개발에 적합한 공식 템플릿이 제공되는 조합이다.
- Rive는 웹 런타임과 State Machine 기반 인터랙티브 애니메이션에 적합하므로, 캐릭터 애니메이션 고도화에 선택적으로 사용할 수 있다.
- FastAPI는 게임 화면이 아니라 팀 상태 저장, 정답 검증, 힌트 기록, 관리자 기능에 사용한다.
- 음악은 판매용 또는 외부 배포용으로 사용할 경우 공연권, 복제, 배포, 음원 사용 허락 등을 별도로 확인해야 한다.
